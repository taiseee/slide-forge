# レイアウトカタログ(完全サンプル)

各レイアウトの「そのまま使える」Markdownサンプル。実レンダリング例は
slide-forgeリポジトリ本体(単体スキルインストールには含まれない)の
`examples/demo-research.md` をビルドすると確認できる。

## title — 表紙

構造: h1(タイトル) → p(サブタイトル) → p(発表者) → p(日付・場所)

```markdown
<!-- _class: title -->
<!-- _paginate: false -->

# 発表タイトル

サブタイトルや一言説明

発表者名

2026年7月8日 · 発表の場
```

## agenda — 目次

構造: h1 + 番号付きリスト(番号は自動で 01, 02, ... と装飾される)。章の途中で再掲するときは、現在位置の項目を `**太字**` にすると背景と番号が強調される。

```markdown
<!-- _class: agenda -->

# 本日の内容

1. **研究背景と課題**
2. 提案手法
3. 実験と結果
4. まとめ
```

## divider — セクション区切り

構造: h1(セクション名) + 任意の p(補足)。番号は h1 に含めて書く

```markdown
<!-- _class: divider -->

# 1. 研究背景と課題

なぜこの問題に取り組むのか
```

## content — 標準スライド

構造: h1 + 箇条書き(ネスト1段まで)/段落/blockquote(強調ボックス)

```markdown
<!-- _class: content -->

# 研究背景

- 主張のポイント1
  - 補足説明
- 主張のポイント2
- **課題**: 強調したいこと

> 結論やメッセージはblockquoteで強調ボックスにできる
```

## two-column — 2段組

構造: h1(全幅) + 本文(左→右へ自動流し込み)。h2 で小見出しを立てられる。
列の切れ目は自動なので、切れ目を制御したいときは comparison を使う。

```markdown
<!-- _class: two-column -->

# 関連研究の整理

## 自動評価指標

- 項目1
- 項目2

## 人手評価

- 項目1
- 項目2
```

## image-right / image-left — 本文+画像

構造: h1 + 本文(箇条書き・段落) + 画像1枚(単独の段落)。
画像は右(左)側の固定領域に自動配置・contain される。

```markdown
<!-- _class: image-right -->

# 提案手法の構成

- ポイント1
- ポイント2

**補足**: 強調したい一言

![](figs/architecture.png)
```

## image-full — 図が主役

構造: h1 + 画像1枚 + 任意の注釈1行(中央寄せ・小さめ表示になる)

```markdown
<!-- _class: image-full -->

# 評価スコアの分布

![](figs/result-chart.png)

図から言えることを1行で
```

## comparison — 対比2列

構造: h1 + (h2 + 内容) × 2。h2 がカード風の列見出しになり、2つ目の h2 から右列に切り替わる。

```markdown
<!-- _class: comparison -->

# 既存手法との比較

## 既存手法

- 特徴1
- 特徴2

## 提案手法

- 特徴1
- 特徴2
```

## table — 表が主役

構造: h1 + テーブル + 任意の p(注記、中央寄せになる)

```markdown
<!-- _class: table -->

# 評価データセット

| データセット | 画像数 | ドメイン |
|---|---|---|
| COCO-Eval | 5,000 | 一般物体 |
| DrawBench | 200 | プロンプト網羅 |

表の読み方や補足を1行で
```

## takeaway — メッセージ強調

構造: h1(伝えたい一言) + 任意の p(補足)。1枚1メッセージ。グラデーション背景・白文字。

```markdown
<!-- _class: takeaway -->

# 局所特徴の導入だけで、相関は 0.54 → 0.69 に向上する

補足の一言(根拠や条件)
```

## summary — まとめ

構造: h1 + 箇条書き(トップレベルがカードになる。ネストはカード内の通常リスト)

```markdown
<!-- _class: summary -->

# まとめ

- 提案内容の要約
  - 補足
- 結果の要約
- 今後の課題: 一言
```

## end — 終了スライド

```markdown
<!-- _class: end -->
<!-- _paginate: false -->

# ご清聴ありがとうございました

質疑応答へ

contact@example.com
```

## experiment — 実験結果(research スキン)

構造: h1 + テーブル(数値は右寄せされる) + blockquote(実験条件・注記)。
最良値は `**太字**` で示す。

```markdown
<!-- _class: experiment -->

# 実験結果: 人手評価との相関

| 手法 | データセットA | データセットB |
|---|---|---|
| ベースライン | 0.542 | 0.481 |
| **提案手法** | **0.687** | **0.634** |

> 条件: 評価指標・モデル・サンプル数などをここに書く。
```

## math — 数式中心(research スキン)

構造: h1 + 導入1行 + `$$...$$`(拡大表示される) + 記号説明の箇条書き

```markdown
<!-- _class: math -->

# スコアの定式化

局所スコアと大域スコアを重み付き和で統合する:

$$
S(x, t) = \lambda \cdot s_{\mathrm{local}}(x, t) + (1 - \lambda) \cdot s_{\mathrm{global}}(x, t)
$$

- $s_{\mathrm{local}}$: 局所スコア、$s_{\mathrm{global}}$: 大域スコア
- $\lambda$: 統合係数
```

## references — 参考文献

構造: h1 + 番号付きリスト(小さめフォント・詰めて表示される)。全スキンで使える。

```markdown
<!-- _class: references -->

# 参考文献

1. Author, A. et al.: Paper Title, Conference, Year.
2. Author, B. et al.: Paper Title, Journal, Year.
```

## timeline — 縦タイムライン

構造: h1 + ol(各 li が「**時期** 内容」)。縦の罫線+ドットで表示される。項目が多いときや説明が短いときに使う。

```markdown
<!-- _class: timeline -->

# 導入ロードマップ

1. **2026 Q3** パイロット導入(30名)
2. **2026 Q4** 効果測定とテンプレート拡充
3. **2027 Q1** 全社展開(500名)
```

## timeline-h — 横タイムライン

構造: h1 + ol(各 li が「**時期** 内容」、2〜5項目)。横罫線+ドットの上に時期、下に説明が並ぶ。

```markdown
<!-- _class: timeline-h -->

# 全社展開スケジュール

1. **2026 Q3** パイロット導入。営業部門30名で試験運用
2. **2026 Q4** 効果測定。工数削減とテンプレート利用率を評価
3. **2027 Q1** 全社展開。500名へロールアウト
```

## steps — 横並びプロセス

構造: h1 + ol(各 li が「**手順名** 説明」、2〜4項目)。大きな番号(01, 02, ...)が自動付与される。時期ではなく順序を示すときに使う。

```markdown
<!-- _class: steps -->

# 導入プロセス

1. **現状調査** 各部門の資料作成フローを棚卸しする
2. **テンプレート整備** 全社共通レイアウトを設計する
3. **試験運用** パイロット部門で運用し改善を回す
```

## columns — 特徴の列並び

構造: h1 + ul(各 li が「**見出し** 説明」、2〜4項目)+ 任意の p(注記)。並列の特徴・強みをテキストで見せる。数値なら kpi を使う。

```markdown
<!-- _class: columns -->

# サービスの特徴

- **内容に集中** 見た目の調整はテンプレートが担う
- **品質の担保** 誰が作っても崩れないレイアウト
- **資産の活用** Markdownベースで移行が容易
```

## spec — ラベル+値の一覧

構造: h1 + ul(各 li が「**ラベル** 値」)。会社概要・仕様・実験条件などのペア情報を罫線区切りで並べる。

```markdown
<!-- _class: spec -->

# サービス概要

- **サービス名** slide-forge
- **提供形態** 社内基盤(オンプレミス)
- **出力形式** HTML / PDF / PNG
```

## faq — Q&A

構造: h1 + (h2(質問)+ 段落(回答))× 2〜3。Q/A の記号は自動付与される。

```markdown
<!-- _class: faq -->

# よくある質問

## デザインをカスタマイズできますか

配色スキンを追加できます。レイアウト自体の改変は受け付けません。

## 既存資料はどうなりますか

移行ツールで段階的にMarkdown化します。
```

## matrix — 2×2マトリクス

構造: h1 + (h2 + 内容)× 4。**象限の順序は 左上 → 左下 → 右上 → 右下**(3つ目のh2から右列に移る)。SWOT・ポジショニング整理に使う。

```markdown
<!-- _class: matrix -->

# 導入判断の整理(SWOT)

## 強み

- 項目

## 弱み

- 項目

## 機会

- 項目

## 脅威

- 項目
```

## gallery — 画像の横並び比較

構造: h1 + **画像2〜3枚を1つの段落に**(同じ行に並べる)+ 任意の注釈1行。

```markdown
<!-- _class: gallery -->

# 生成結果の比較

![](figs/baseline.png) ![](figs/ours.png)

左: ベースライン、右: 提案手法
```

## stat — 数字1つを全面に

構造: h1(数字+単位のみ) + p(何の数字か) + 任意の p(補足)。数字が複数あるなら kpi を使う。

```markdown
<!-- _class: stat -->

# 75%

資料作成時間の削減率(パイロット実測)

週4時間 → 週1時間、営業部門30名の平均
```

## profile — 人物紹介

構造: h1 + h2(名前) + ul(**ラベル** 値 の経歴リスト) + 写真1枚(単独の段落)。写真は円形にトリミングされ左に配置される。

```markdown
<!-- _class: profile -->

# 発表者紹介

## 発表 太郎

- **所属** ◯◯大学大学院 △△研究室(M2)
- **研究テーマ** 画像生成の自動評価
- **連絡先** contact@example.com

![](figs/portrait.jpg)
```

## kpi — 数値ハイライト(business スキン)

構造: h1 + ul(各 li が「**数値** ラベル」、2〜4項目) + 任意の p(注記)。
li がカードとして横並びになり、太字部分が大きな数値表示になる。

```markdown
<!-- _class: kpi -->

# 直近四半期の実績

- **120%** 売上成長率(YoY)
- **3.5万人** 月間アクティブユーザー
- **12社** 新規導入企業

注記や集計条件を1行で
```

## quote — 引用(顧客の声・先行研究の一文)

構造: h1(省略可) + blockquote(引用本文) + p(出典・話者)。
引用が大きく表示される。

```markdown
<!-- _class: quote -->

# 試験導入部門の声

> 資料の見た目を整える時間がほぼゼロになり、提案の中身を考える時間が倍になった。

営業部 マネージャー(試験導入アンケートより)
```

## code — コード中心

構造: h1 + コードブロック(言語指定でハイライト) + 任意の p(注釈1行)。行数の多いコードは載せず、見せたい部分だけ抜粋する。

````markdown
<!-- _class: code -->

# スコア計算の実装

```python
def score(image, text, lam=0.6):
    return lam * local(image, text) + (1 - lam) * global_(image, text)
```

CLIP埋め込みは既存モデルを再利用し、追加学習は統合係数のみ
````

## checklist — チェックリスト

構造: h1 + ul(トップレベル各項目に ✓ が自動付与)。要件・準備事項・達成項目の確認に使う。

```markdown
<!-- _class: checklist -->

# パイロット開始前チェックリスト

- 対象部門と参加メンバーの確定
- 効果測定の指標定義
- 研修資料の準備
```

## team — 複数人の紹介

構造: h1 + ul(各 li が「![](写真) **名前** 役割・説明」、2〜4人)。写真は円形トリミングで横並びになる。1人だけなら profile を使う。

```markdown
<!-- _class: team -->

# 推進体制

- ![](figs/yamada.jpg) **山田** プロジェクトリード。全体統括
- ![](figs/sato.jpg) **佐藤** テンプレート設計
- ![](figs/suzuki.jpg) **鈴木** 導入支援・研修
```

## lead — 導入メッセージ(明るい面)

構造: h1(メッセージ) + 任意の p(補足)。章の間の問いかけや導入に使う。濃色面でよければ takeaway。

```markdown
<!-- _class: lead -->

# 生成画像の「良さ」を、人手に頼らず測れるか。

本研究が取り組む問い
```

## funnel — 絞り込みの段階

構造: h1 + ol(各 li が「**段階名 数値** 補足」、3〜5段)。上から狭まり、最終段が濃色で強調される。

```markdown
<!-- _class: funnel -->

# 商談ファネル

1. **問い合わせ 120件** 展示会・Web経由
2. **商談化 64件** 初回ヒアリング実施
3. **提案 31件** 個別提案書を提出
4. **受注 12件** 受注率 38.7%
```

## flow — 矢印つき横フロー

構造: h1 + ol(各 li が「**要素名** 説明」、2〜4項目)。箱と→でプロセスや構成を示す。番号で見せたいときは steps。

```markdown
<!-- _class: flow -->

# サービス提供フロー

1. **執筆** Markdownで内容だけを書く
2. **生成** テンプレートを適用してビルド
3. **検証** はみ出しを自動チェック
```

## agenda-grid — 目次の2列版

構造: agenda と同じ(h1 + 番号付きリスト)。項目が7個以上のときに使う。**番号は左→右→次の行の順**に振られる。`agenda` と同様、現在位置を `**太字**` にすると強調される。

```markdown
<!-- _class: agenda-grid -->

# 本日の内容

1. **現在の項目**
2. 項目
(8項目程度まで)
```

## pyramid — 階層構造

構造: h1 + ol(各 li が「**階層名** 説明」、3〜4段)。上から下へ広がり、頂点(1つ目)が濃色で強調される。ビジョン→戦略→施策などに使う。

```markdown
<!-- _class: pyramid -->

# 目指す姿

1. **ビジョン** 内容だけを考えれば良い組織
2. **戦略** テンプレートと自動検証の全社基盤
3. **施策** パイロット導入 → 研修 → 全社展開
```

## venn — 2円ベン図

構造: h1 + ul(li がちょうど3つ:「**左の集合** 説明」「**右の集合** 説明」「**共通** 一言」)。**3つ目(共通)のテキストは短く**(ラベル+数語まで)。

```markdown
<!-- _class: venn -->

# 提案の位置づけ

- **既存の自動評価** 大域特徴・追加学習なし
- **人手評価** 細部の品質・高コスト
- **提案手法** 両者の橋渡し
```

## before-after — ラベル付き画像比較

構造: h1 + (h2(ラベル) + 画像1枚)× 2 + 任意の p(注記)。改善の対比を見せる。ラベル不要なら gallery。

```markdown
<!-- _class: before-after -->

# 評価の改善例

## ベースライン

![](figs/before.png)

## 提案手法

![](figs/after.png)

同一プロンプトに対するスコアの説明性が向上
```

## logos — ロゴ・実績の並び

構造: h1 + ロゴ画像を1つの段落にまとめる(3〜8個) + 任意の p(注記)。ロゴはグレースケール・統一の高さで表示される。

```markdown
<!-- _class: logos -->

# 導入企業・パートナー

![](logos/a.png) ![](logos/b.png) ![](logos/c.png) ![](logos/d.png)

2026年6月時点
```

## plans — 料金プラン・パッケージ比較(business スキン)

構造: h1 + (h2(プラン名) + p(**価格** 補足) + ul(内容))× 2〜3。価格は太字にすると大きく表示される。

```markdown
<!-- _class: plans -->

# 提供プラン

## ライト

**月5万円** / 部門

- テンプレート利用
- メールサポート

## スタンダード

**月15万円** / 部門

- ライトの全機能
- 移行ツール・研修
```

## gantt — 簡易ガントチャート

構造: h1 + テーブル + 任意の p(注記)。**1列目はタスク名、以降の列は期間**。バーを引く期間のセルに任意の1文字(x 推奨)を入れ、引かない期間は空にする(文字は透明化されバーになる)。期間は6列程度まで。

```markdown
<!-- _class: gantt -->

# 研究計画

| タスク | 7月 | 8月 | 9月 | 10月 |
|---|---|---|---|---|
| 追加分析 | x | x |  |  |
| 拡張実験 |  | x | x |  |
| 論文執筆 |  |  | x | x |

バーの期間は暫定
```

## org — 体制図・組織図

構造: h1 + h2(トップ・全体名) + ul(各 li が「**部門名** 説明」、2〜4個)。トップから各ボックスへ接続線が引かれる。人の顔を見せたいときは team。

```markdown
<!-- _class: org -->

# 実施体制

## 全社導入プロジェクト

- **企画チーム** 要件定義・効果測定
- **開発チーム** テンプレート整備
- **導入支援チーム** 研修・問い合わせ対応
```

## title-visual — 写真・キービジュアル入り表紙

構造: title と同じ(h1+サブタイトル+発表者+日付)。**同スライドに `![bg right:45%](画像)` を書く**とテキストが左半分に収まる。写真はそのまま(cover)、図版なら `fit` を付ける。

```markdown
<!-- _class: title-visual -->
<!-- _paginate: false -->

![bg right:45%](figs/keyvisual.jpg)

# 発表タイトル

サブタイトル

発表者名

2026年7月8日 · 発表の場
```

```markdown
<!-- _class: exec-summary -->

# エグゼクティブサマリー
```

```markdown
<!-- _class: content-lead -->

# 考察

- 根拠1

- **限界**: 注意点
```

## cheatsheet — チートシート(lecture スキン)

構造: h1 + ul(各 li が「`コマンド` 説明」、6〜12個) + 任意の p(注記)。2列の密なグリッドに左→右・上→下の順で流れる。説明は短く1行に収める。

```markdown
# コマンド早見表
```

## quiz — 演習・クイズ(lecture スキン)

構造: h1 + blockquote(問題文。行頭に Q ラベルが付く) + ol(選択肢。A〜D のチップ付きで表示される)。解答は次の `answer` スライドに分ける。

```markdown
<!-- _class: quiz -->

# 演習

> `git branch feature` を実行した直後、リポジトリに新しく作られるものは
> どれか。

1. コミットオブジェクトのコピー一式
2. 現在のコミットの SHA を書いたファイル refs/heads/feature
3. 作業ツリー全体のスナップショット
4. 何も作られない
```

## answer — 解答・解説(lecture スキン)

構造: h1 + p(結論。**strong** で正解を示す。行頭に A ラベルが付く) + ul(解説 2〜4項目)。`quiz` とペアで使う。

```markdown
<!-- _class: answer -->

# 解答

**B. refs/heads/feature という参照ファイルが1つ作られるだけ**

- ブランチ作成はファイル1つの書き込みなので一瞬で終わる
- コミットの実体(オブジェクト)は一切コピーされない
```

## code-focus — コード解説(lecture スキン)

構造: h1 + コードブロック(①②③ の丸数字コメントを行末に付ける) + ol(各番号の解説)。コードが左、解説が右に並ぶ。コードは長すぎると切れるため 15行程度まで。

````markdown
<!-- _class: code-focus -->

# cat-file でオブジェクトを辿る

```shell
$ git cat-file -p HEAD          # ①
tree a1b2c3d4

$ git cat-file -p a1b2c3d4      # ②
100644 blob 5d6e7f8a  README.md
```

1. **commit** tree と parent への参照+メタデータを持つ
2. **tree** ファイル名と blob/tree を対応づける表
````

## code-compare — 良い例・悪い例のコード対比(lecture スキン)
